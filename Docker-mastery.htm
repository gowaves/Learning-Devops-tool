2 The Docker Registery

       We have our image that we just built and that image has a unique SHA hash

       We can use something like Docker push to take those image layers

and push them up to a registry.

       You can use one of many registries, like I mentioned, But once it's there, you

can then go to another machine and do a Docker pull, which will pull an identical copy of that image down on that machine.

       Just like we had built it on the first machine.

 

3 The Docker Container

       Lets your server has the docker running in the background that know as the docker engine and the engine on this single machine will orchestrate or manage all the different things that need to happen in order to

       download images, verify that they're the exact thing that was meant to be downloaded, and then finally run them.

       Then we got the image, we just downloaded with a Docker pull, and then we're gonna run a command.

       And in this case, we're running that Python app in its own container. That container is known as a namespace.

        if we run that Docker, run command again with our image name or image hash, then we get an identical container running side by side.

       Now these two processes are isolated from each other. So the Python app in container one can't see the Python app in container two.

       And if you change a file in container two, container one won't see that file by default.

       If we added a second server with a second instance of the Docker Engine running and did the same thing,

       we could have a highly available setup with our application running many times for redundancy or better performance or whatever our reason might be for running more than one of them.
